# Documentation Source of Truth

When determining the factual state of this project, resolving conflicts between documents, or making architectural decisions, the following hierarchy of authority applies.

When two documents disagree, the higher document wins.

## Authority Hierarchy

1. **AuditIQ Master Project Bible**
2. **Approved Blueprint Documents**
3. **Stable Release Tags**
4. **Repository Source Code**
5. **Validation Reports & Audits**
6. **Governance Rules & Data Contracts**
7. **Architecture Documents**
8. **Roadmaps**

Repository code and Git history strictly override any assumptions, simulated capabilities, or historical chat memory. If a feature exists in the code but contradicts the Master Project Bible, the Bible is the authoritative source of intent, and the code represents a deviation that should be audited or corrected (unless a newer stable tag formally supersedes it).
