# AuditIQ Technical Baseline V1

_Baseline captured from the current repository state at the latest commit._

## 1. Purpose

AuditIQ is a Next.js procurement-audit application that runs a deterministic three-way analysis flow across:

- Purchase Order
- Goods Receipt Note
- Vendor Invoice

This baseline documents the current implementation as it exists today, including repository structure, processing flow, data models, storage, technical debt, and safe modification zones.

## 3. Current Application Topology

The app is a small Next.js App Router project with three primary routes:

- `/` — dashboard summary
- `/upload` — upload and analysis orchestration
- `/results` — detailed results view

The application is client-heavy around upload/results because browser APIs such as `sessionStorage` and PDF parsing are used directly in those pages.

## 4. Upload Flow

The implemented upload flow is in `src/app/upload/page.tsx`.

1. The user selects one file each for PO, GRN, and Invoice.
2. The page validates that all three files are present.
3. The filenames are classified using `classifyDocument(file.name)`.
4. Each PDF is read as text using `readPdfText(file)`.
5. The extracted text is passed to `extractDocumentData(documentType, documentText)`.
6. The three extracted document objects are compared with `matchDocuments(...)`.
7. Exceptions are generated by `detectExceptions(...)`.
8. Financial exposure is computed with `calculateFinancialExposure(...)`.
9. Risk is computed with `assessRisk(...)`.
10. Recommendations are generated with `generateRecommendations(...)`.
11. Explainability is generated with `generateExplainability(...)`.
12. The assembled analysis snapshot is serialized to `sessionStorage` under `auditIQAnalysis`.
13. The page routes to `/results`.

### Upload validation behavior

- If any of the three files is missing, analysis stops and a validation message is shown.
- Duplicate invoice detection is wired into the engine, but the current upload flow passes an empty `existingInvoiceNumbers` array.

## 5. PDF Reading Flow

PDF text extraction is implemented in `src/lib/pdfTextReader.ts`.

### Flow

1. The module lazily imports `pdfjs-dist/legacy/build/pdf.mjs`.
2. It configures `pdfjs.GlobalWorkerOptions.workerSrc` to `/pdf.worker.min.mjs`.
3. The uploaded `File` is converted to an `ArrayBuffer`.
4. PDF.js opens the document from raw bytes.
5. Each page is read with `getTextContent()`.
6. Text items are joined into page-level strings and then into a full document string.
7. If any error occurs, the reader returns an empty string.

### Operational characteristics

- The implementation is browser-side only.
- The worker file is served from `public/pdf.worker.min.mjs`.
- PDF errors are swallowed and degrade to empty text, which makes extraction fall back to fixture values.

## 6. Classification

Classification is implemented in `src/lib/classifier.ts`.

### Behavior

- `po` in the filename → `Purchase Order`
- `grn` in the filename → `Goods Receipt Note`
- `inv` or `invoice` in the filename → `Vendor Invoice`
- anything else → `Unknown`

### Characteristics

- Classification is deterministic and synchronous.
- It uses filename heuristics only.
- It does not inspect file contents, metadata, or layout.

### Implication

The current classifier is good enough for a controlled MVP but is fragile outside naming conventions.

## 7. Extraction

Extraction is implemented in `src/lib/extractor.ts`.

### Current extraction model

The extractor accepts:

- `documentType`
- `documentText`

It returns one of the following:

- a structured document object
- a fallback fixture object
- `null` for unknown document types

### What it extracts today

Each supported document shape contains:

- `vendor`
- `documentNumber`
- `date`
- `quantity`
- `unitPrice`
- `amount`

### How extraction works

- The module contains built-in fixture data for PO, GRN, and Invoice.
- It attempts regex-based extraction from the PDF text for numeric fields.
- It looks for labeled values such as:
  - quantity / qty
  - unit price / price per unit / unit cost / rate
  - amount / total amount / invoice amount / grand total / total
- If all three numeric fields are found, they override the fixture numeric values.
- If text parsing fails, the fixture values are returned unchanged.

### Current limitation

Only the numeric fields are partially text-aware. Vendor, document number, and date remain fixture-driven.

## 8. Matching

Matching is implemented in `src/lib/matcher.ts`.

### Compared fields

- quantity
- unit price
- amount

### Matching rule

A field matches only when:

- PO value is present
- GRN value is present
- Invoice value is present
- all three values are exactly equal

### Output

The matcher returns:

- `quantityMatch`
- `priceMatch`
- `amountMatch`

Each field includes:

- `matched`
- `po`
- `grn`
- `invoice`

### Current limitation

The matcher uses exact equality only. There is no tolerance band, currency normalization, rounding logic, or line-item reconciliation.

## 9. Exception Engine

Exception detection is implemented in `src/lib/exceptionEngine.ts`.

### Current exception types

- Quantity Mismatch
- Price Variance
- Missing Invoice
- Missing GRN
- Duplicate Invoice

### Detection rules

- quantity mismatch is raised when quantity matching fails
- price variance is raised when price matching fails
- missing invoice is raised when no invoice object is available
- missing GRN is raised when no GRN object is available
- duplicate invoice is raised when the invoice document number exists in `existingInvoiceNumbers`

### Severity

- all exceptions are emitted with severity `High`

### Current limitation

Duplicate invoice detection is only effective when the caller provides an invoice history list. The current upload flow does not provide one, so the path is present but not operational in practice.

## 10. Risk Engine

Risk assessment is implemented in `src/lib/riskEngine.ts`.

### Scoring model

- Base score = `exceptions.length * 10`
- Exposure adjustment:
  - `> 25000` → `+30`
  - `> 10000` → `+20`
  - `> 5000` → `+10`
  - otherwise `+0`
- Final score is capped at `100`

### Risk levels

- `0–25` → Low
- `26–50` → Medium
- `51–75` → High
- `76–100` → Critical

### Characteristics

- The model is deterministic and rule-based.
- It is driven by exception count and total exposure only.
- It does not weigh exception type, vendor history, or document confidence.

## 11. Recommendation Engine

Recommendation generation is implemented in `src/lib/recommendationEngine.ts`.

### Behavior

The engine builds a deduplicated list of remediation actions from:

- exception types
- risk level
- total exposure

### Current recommendations

- Review received quantity and reconcile with purchase order.
- Review invoice pricing and obtain approval for variance.
- Request invoice from vendor before payment.
- Confirm goods receipt before processing invoice.
- Block payment and investigate duplicate invoice.
- Escalate to finance manager for review.
- Hold payment until discrepancy is resolved.

### Characteristics

- Deterministic and rule-based
- Duplicate recommendations are removed
- No external policy engine or AI service is used

## 12. Explainability Engine

Explainability is implemented in `src/lib/explainability.ts`.

### Purpose

The engine turns the computed analysis into human-readable narrative output.

### Inputs

- match result
- exceptions
- financial exposure
- risk result
- recommendations
- extracted document objects

### Outputs

- `summary`
- `explanations[]`

### What it explains

- why a quantity mismatch was flagged
- why a price variance was flagged
- why documents are missing
- why duplicate invoice risk exists
- how total exposure was derived
- why the risk score and level were selected
- why each recommendation was generated

### Characteristics

- Template-driven and deterministic
- No AI model calls
- No external dependency

## 13. Storage Architecture

AuditIQ currently uses browser `sessionStorage` as its persistence layer.

### Storage key

- `auditIQAnalysis`

### Writer

- `src/app/upload/page.tsx`

### Readers

- `src/app/results/page.tsx`
- `src/app/page.tsx`

### Stored payload

The stored object is a serialized `AnalysisResult` snapshot containing:

- `files`
- `classifications`
- `extractedData`
- `matchResult`
- `exceptions`
- `financialExposure`
- `risk`
- `recommendations`
- `explainability`

### Storage characteristics

- tab-scoped
- browser-local
- temporary
- JSON-based
- no server persistence
- no multi-user history

### UI behavior on missing storage

- the dashboard falls back to an empty analysis state
- the results page falls back to a sample analysis object

## 15. Current Technical Debt

The current codebase is intentionally simple, but several pieces are still MVP-grade.

### Highest-priority debt

- Filename-based classification is brittle.
- Extraction is still fixture-first, not document-first.
- Duplicate invoice detection lacks a persistent invoice history.
- Session storage is not durable enough for audit retention.
- PDF parse failures silently degrade into empty text.

### Structural debt

- Upload orchestration lives in a single client page.
- The pipeline is tightly coupled to browser execution.
- There are no automated tests in the repository.
- There is no schema validation layer for stored analysis snapshots.
- Results fallback data can mask storage or extraction issues during manual review.

### Logic debt

- Matching is exact only.
- Risk scoring is simplistic and exposure-threshold driven.
- Recommendation logic is fixed-rule text, not policy-driven.
- Explainability is templated rather than evidence-rich.

### Product debt

- No authentication or access control
- No backend API
- No durable audit history
- No OCR or document intelligence

## 16. Safe Modification Zones

These are the lowest-risk places to evolve the product without changing the core analysis semantics.

### Generally safe

- `src/app/layout.tsx`
  - metadata, document shell, global wrapper structure
- `src/app/globals.css`
  - theme, spacing, and visual styling
- `src/app/page.tsx`
  - dashboard copy, layout, and non-semantic presentation
- `src/app/results/page.tsx`
  - result-card layout, presentation, and labeling
- `README.md` and `docs/*`
  - documentation and screenshots
- `public/*` assets
  - icons, illustrations, and worker file placement

### Safe with light coordination

- adding presentational components under a future `src/components` directory
- adding display-only fields to the results page
- refining recommendation wording without changing the engine rules
- improving empty-state messages and loading states

### Areas that should be treated as high-caution

- `src/app/upload/page.tsx`
- `src/lib/classifier.ts`
- `src/lib/extractor.ts`
- `src/lib/matcher.ts`
- `src/lib/exceptionEngine.ts`
- `src/lib/financialExposure.ts`
- `src/lib/riskEngine.ts`
- `src/lib/recommendationEngine.ts`
- `src/lib/explainability.ts`
- `src/lib/pdfTextReader.ts`

These files define the core pipeline and should be changed carefully because they directly affect the analysis semantics and stored snapshot format.

## 17. Identifier Extraction Readiness

### Current readiness level

**Partial.**

AuditIQ is ready to ingest PDF text, but it is not yet ready for robust identifier extraction at production quality.

### What is already in place

- browser-side PDF text extraction exists
- the extractor can inspect raw text
- numeric labeled values can be parsed from text when present
- the analysis snapshot already has slots for document metadata such as `vendor`, `documentNumber`, and `date`

### What is still missing

- true OCR for scanned documents
- document-layout parsing
- table and line-item reconstruction
- confidence scoring
- identifier normalization and deduplication
- vendor/document-number/date extraction from real document content
- persistent historical lookup for duplicate invoices

### Practical interpretation

The current pipeline is a good scaffold for identifier extraction, but identifier intelligence is still mostly simulated through fixtures. The architecture is ready for the next step, but the extraction quality is not yet production-grade.

## 18. Baseline Summary

AuditIQ V1 is a small, deterministic, browser-local Next.js application. Its strongest current qualities are clarity, traceability, and low operational complexity. Its main constraints are fixture-driven extraction, filename-based classification, and temporary session storage.

The core pipeline is already cleanly separated into stages, which makes the repository a strong candidate for incremental hardening in future versions.
